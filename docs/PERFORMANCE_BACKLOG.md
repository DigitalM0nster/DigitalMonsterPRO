# Бэклог производительности (CPU / GPU)

**Уже сделано в коде:** CPU п.2 и п.3 — `PhoneModel.jsx` (цепочка `else if` по ширине канваса; один `setTimeout` на отложенное скрытие по скроллу + сброс таймера на портфолио/скрытых URL). GPU п.6 — упрощена декоративная сфера в `Case4Model.jsx` при `graphicsTier === "low"`. **CPU п.5** — ранние выходы в `BackgroundScheme`, `RoofModel`, `LinesModel` (Case2); убран `setState` из шейдерного фона; `MyEnvironment` — одна ветка света за кадр + guard; `CanvasRoutes` — `Sparkles` только на главной, ранний выход в `useFrame` после `rendered` и безопасные проверки ref.

Зафиксировано по запросу: план по пунктам CPU 6 и проверка GPU 6.

## CPU — пункт 5: `useFrame` в дочерних компонентах

**Суть:** Любой `useFrame` в Drei/React Three Fiber выполняется каждый кадр, пока компонент смонтирован, даже если объект визуально не участвует в сцене.

**Сделано (основной сценарий Case2 + сцена):**

1. `case2/case2Constants.js` — общий `CASE2_PATH`.
2. `BackgroundScheme`, `RoofModel`, `LinesModel` — `return`, если не `/portfolio/02` или корень Case2 `visible === false`; в фоне убрано обновление React state каждый кадр.
3. `MyEnvironment` — проверка `directionalLightRef`, объединены маршруты с одинаковым светом, исправлено условие скролла на `/portfolio` (`> 0.35 && <= 0.9`).
4. `CanvasRoutes` — условный рендер `Sparkles` только для главной (`split("/")[1] === ""`).

**Дальше по желанию:** `EffectComposerComponent` (длинная цепочка `if` по страницам), `Float` из Drei, другие эффекты — смотреть профайлер.

**Критерий готовности:** в React Three Fiber Profiler / замере времени кадра меньше колбэков на кадр на неактивных маршрутах.

## CPU — пункт 6: загрузка моделей (разнесение по маршрутам)

**Суть:** Сейчас в `CanvasRoutes` одновременно смонтированы все кейсы, About, Contacts, Phone — растёт память, время компиляции шейдеров и пик при первом кадре.

**Что сделать по шагам:**

1. Составить таблицу: какая модель нужна на каком `pathname` (с запасом на переходы, если нужна предзагрузка соседнего кейса).
2. Ввести ленивую подгрузку: `React.lazy` + `Suspense` для групп моделей **или** условный рендер `{currentPage.startsWith('/portfolio/04') && <Case4Model />}` с возможным `key` для размонтирования.
3. Учесть `setRendered` / прелоадер: не ломать ожидание «первого кадра готов».
4. Протестировать: холодный заход на `/`, затем навигация по всем портфолио-кейсам — нет ли белого кадра без `Suspense` fallback.

**Критерий готовности:** меньше одновременно загруженных GLB/текстур; быстрее TTI на слабых устройствах.

## GPU — пункт 6: прозрачность и overdraw (проверка)

**Проверено в коде:**

- В `Case4Model.jsx` декоративная сфера: `wireframe` + `transparent` + высокая сетка `sphereGeometry [1, 26, 13]` — типичный источник лишнего fillrate (много линий/смешения).
- **Сделано:** для `store.graphicsTier === 'low'` сфера переведена на меньшую сегментацию, без wireframe и без прозрачности (сплошной тёмный материал), с `key` по tier для корректной смены геометрии.

**Дополнительно на будущее:** полупрозрачные плоскости и фоновый composer.

**Постобработка (сделано):** условное отключение Distortion/Blur когда параметры ≈ 0; `stencilBuffer={false}`; dev-профилирование — см. [`docs/POSTPROCESSING_PERFORMANCE.md`](./POSTPROCESSING_PERFORMANCE.md) (`?perf=1`, `?ppDebug=1`, `?ppBloomOnly=1`).
